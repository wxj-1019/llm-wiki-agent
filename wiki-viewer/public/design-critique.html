Wiki-Viewer Frontend · 设计评审报告

Reviewer: OD · Critique skill · 2026-05-06
整体判断：设计方向清晰、系统成熟，但组件一致性存在「最后一公里」缺口
1. 哲学一致性 7 Strong

Apple 暖纸设计方向明确，--bg-primary / --text-primary 语义 token 体系完整，glassmorphism 与 pill 按钮贯穿全站。但方向在边缘页面出现漂移：CrawlerPage 的按钮混用 rounded-xl 与 rounded-lg，SettingsPage 的 tab active 状态在 core tabs 用 text-apple-purple 而 source tabs 用 text-[var(--text-primary)]，同一组件内出现两种强调色语义。Sidebar 的 active 状态在 collapsed 时用 bg-apple-blue/12、expanded 时用 /8，虽然意图区分状态，但视觉上形成了两套重量标准。

2. 视觉层级 7 Strong

三级文本 token（primary / secondary / tertiary）划分清晰，Header 的 backdrop-blur-xl 与固定层级创造了稳定的「天花板」锚点。HomePage 的 StatCard 采用 icon + value + label 三层结构，信息节奏良好。问题出在 button 层级的泛滥：.apple-button、.apple-button-warm、.apple-button-ghost、.btn-primary、.btn-secondary 五套按钮在同一产品内竞争，用户无法建立稳定的「主按钮 = 蓝色 pill、次按钮 = ghost」心智模型。CrawlerPage 的 save 按钮用自定义 bg-apple-green/10 样式，进一步稀释了层级规则。

3. 细节执行 6 Functional

骨架良好，但存在多处「 tape and string」级别的粗糙。index.css 中 @theme 与 :root 重复定义同一套颜色（如 --apple-blue 出现两次），存在维护隐患。RadarChart 使用硬编码 rgba(10, 132, 255, 0.08) 而非已定义的 --chart-fill CSS 变量。ToastContainer 的 progress bar track 用 bg-[var(--border-default)]，而 border color 映射使用 /20 透明度、background 用 /10，track 强度与边框不匹配。AppleSelect 的 trigger 同时用 Tailwind class shadow-[0_0_0_4px_...] 和 inline style borderColor 设置边框色，属于冗余声明。

4. 功能性 8 Strong

防御性工程做得扎实：Keyboard shortcuts（g h / g b / g g 等）覆盖主要导航；Skip-to-content 链接具备 sr-only focus:not-sr-only 模式；AppleSelect 实现了完整的 listbox 键盘协议（ArrowUp/Down、Home/End、Escape、aria-activedescendant）；Framer Motion 通过 MotionConfig reducedMotion="user" 全局响应 prefers-reduced-motion；Focus trap 在 CommandPalette 搜索弹窗中正确工作。三态主题（light/dark/system）配合 matchMedia 实时同步，FOUC 预防脚本在 index.html 中提前执行。唯一缺口：SettingsPage 的 tab 切换未提供键盘快捷键，且 custom segmented control 未复用 AppleSelect 的成熟键盘逻辑。

5. 创新性 5 Functional

整体属于「高完成度的标准品」。Apple 设计系统本身已是行业标杆，项目对其的还原度较高，但缺少一个让人「lean in」的意外时刻。 RadarChart 的自定义 SVG 实现是亮点，但数据可视化语言未形成体系（与 GrowthTrendChart 的 SVG 线表风格不统一）。Ambient blob 装饰动画（.ambient-blob）存在但未在核心页面大面积使用，停留在「有但不多」的尴尬地带。缺少一个标志性的品牌微交互——例如 Notion 的 loading skeleton 动画或 Linear 的命令行式快捷导航视觉反馈。作为生产级交付，保守是合理的；但作为 showcase，需要一个记忆点。

Keep · 继续保留

  • CSS 语义 token 体系 — --bg-primary/secondary/tertiary 与 --text-primary/secondary/tertiary 的三级分层清晰，不要简化成两级。
  • Framer Motion 页面转场 — 按路由类型区分 scale-fade / horizontal-slide / vertical-slide 三种动画 profile，这是专业级决策。
  • AppleSelect 的 ARIA 完整度 — aria-activedescendant + 方向键导航 + Home/End + Escape 返回焦点，已达到 WCAG 2.1 AA 标准。
  • 三态主题系统 — light / dark / system 配合 localStorage 持久化与 FOUC 预防脚本，机制完整。
  • Glassmorphism Header — backdrop-blur-xl bg-[var(--bg-primary)]/75 的半透明磨砂质感与内容区形成正确的景深关系。

Fix · 必须修复

  • P0 · 合并 button 变体 — 将 .apple-button / .apple-button-warm / .apple-button-ghost / .btn-primary / .btn-secondary 收敛为 2 个语义变体（primary / secondary），删除冗余 class。CrawlerPage 的 save 按钮应使用标准 secondary 样式而非自定义绿色背景。
  • P0 · 清理 CSS 重复定义 — index.css 中 @theme 块与 :root 块对同一颜色的重复定义必须合并，只保留一个来源。
  • P1 · SettingsPage tab active 色统一 — core tabs 与 source tabs 的 active 颜色必须一致（建议统一为 text-apple-blue，与全局强调色对齐）。
  • P1 · CrawlerPage 卡片未使用 apple-card — 将 stats grid 和配置面板迁移到 .apple-card，消除「CrawlerPage 看起来像另一个产品」的视觉断裂。
  • P1 · 统一 border-radius 语义 — 建立规则：交互容器（card / panel）= rounded-2xl，按钮 = rounded-xl，标签/badge = rounded-full，小按钮/输入框 = rounded-lg。当前 CrawlerPage 混用 rounded-xl 与 rounded-lg 无规则。

Quick Wins · 5-15 分钟高回报

  • 替换 RadarChart 硬编码颜色 — 将 rgba(10, 132, 255, 0.08) 改为 var(--chart-fill)，已定义但未使用。
  • 删除 AppleSelect 冗余 borderColor — 移除 trigger 上的 inline style borderColor，Tailwind class 已覆盖。
  • Sidebar active 状态统一透明度 — collapsed / expanded 统一为 bg-apple-blue/10，消除视觉重量差异。
  • ToastContainer progress track 对齐 — 将 track 改为 bg-apple-purple/10 与 toast background 同频，而非 var(--border-default)。
  • 添加一个品牌微交互 — 在 Header 的 logo 或首页的 stat card hover 上添加一个 subtle 的 scale + glow 效果（3 行 CSS），制造记忆点。
